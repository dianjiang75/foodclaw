### Test 2: Max Wait 15 Minutes

| Field | Value |
|-------|-------|
| Endpoint | `GET /api/search?lat=40.7264&lng=-73.9878&max_wait=15&sort=wait_time` |
| Status | `200 OK` |

All returned dishes have wait times in {7, 8, 9, 11} minutes -- all <= 15. 20 dishes returned (total_count=45, so pagination applies). Filter correctly excludes higher-wait restaurants.

### Test 3: Max Wait 10 Minutes (Stricter)

| Field | Value |
|-------|-------|
| Endpoint | `GET /api/search?lat=40.7264&lng=-73.9878&max_wait=10&sort=wait_time` |
| Status | `200 OK` |
| Result | **FAIL** |

**BUG: max_wait filter not enforced.** Returned dishes have wait times {7, 8, 9, 11} -- the value 11 exceeds the max_wait=10 threshold. The filter appears to be ignored or only partially applied. total_count=45 (same as unfiltered), confirming the filter has no effect.

---

### Test 4: Distance Sort

| Field | Value |
|-------|-------|
| Endpoint | `GET /api/search?lat=40.7264&lng=-73.9878&sort=distance&limit=20` |
| Status | `200 OK` |
| Result | **FAIL** |

Distances are ascending (0.10, 0.10, 0.10, 0.12, ...) so the sort order is correct. However, **Raku (0.06 mi) is missing entirely** from the result set despite being the closest restaurant. It appears in `/api/restaurants` and in `/api/search` with `radius=1`, but not in the plain distance sort. This is a data-consistency bug where at least one restaurant is dropped from unfiltered distance-sorted search results.

**Issues:**
- Raku at 0.06 mi absent from 45 total results
- Nearest returned restaurant is Xi'an Famous Foods at 0.10 mi

---

### Test 5: Distance Sort + Radius 1 Mile

| Field | Value |
|-------|-------|
| Endpoint | `GET /api/search?lat=40.7264&lng=-73.9878&sort=distance&radius=1&limit=20` |
| Status | `200 OK` |
| Result | **PASS** |

All returned distances are within 1 mile: {0.06, 0.1, 0.12, 0.15, 0.24, 0.48, 0.52}. Raku at 0.06 mi IS present here (unlike Test 4). Distances are ascending. total_count=43 (correctly fewer than unfiltered 45). 20 dishes returned with limit applied.

---

## Instance 1 Score: 6/10

Three tests pass cleanly. Two failures: max_wait=10 filter is broken (dishes with 11-min wait leak through), and distance sort without radius drops the closest restaurant (Raku). The max_wait bug is a functional regression; the missing-restaurant bug is a data-consistency issue.

## Overall Persona Score: 6/10

The speed-focused user experience is undermined by a broken wait-time filter. Speedy Sam's core use case -- finding the fastest nearby food -- does not work as expected. Distance sorting and radius filtering are functional, but the wait-time filtering (the most critical feature for this persona) needs immediate attention.
